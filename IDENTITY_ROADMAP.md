# LightCode — Identity Roadmap

Este roadmap define cómo aterrizar la identidad oficial de LightCode desde la situación actual hasta una versión coherente y releaseable.

La dirección ya elegida es:

- **categoría**: persistent coding system
- **metáfora**: Memory Atlas
- **superficie principal**: Atlas Field
- **paleta base**: Void Black

---

## Objetivo general

Hacer que LightCode se perciba como un producto propio con una interfaz basada en memoria relacional, no como un fork temático ni como un chat tool genérico.

Resultado buscado:

> el usuario debe sentir que navega un atlas de trabajo y memoria, no solo una lista de conversaciones.

---

## Principios rectores

### 1. El grafo no es decoración
El Atlas Field es la superficie principal del producto.

### 2. La memoria debe ser visible
Working memory, anchors, signals y continuidad deben sentirse estructurados.

### 3. La identidad no depende del logo
Depende de:
- color
- vocabulario
- composición
- relación entre paneles
- semántica del campo

### 4. No romper legibilidad por estilo
La metáfora cósmica sirve solo si mejora percepción, orientación y navegación.

# Roadmap por fases

## Fase 1 — Cerrar el sistema de identidad
### Objetivo
Congelar la dirección para que implementación y diseño no sigan abriendo ramas conceptuales.

### Entregables
- identidad verbal cerrada
- paleta cerrada
- metáfora cerrada
- composición principal cerrada
- vocabulario visible cerrado

### Estado deseado
No más exploración amplia. Solo refinamiento.

---

## Fase 2 — Reescribir la documentación de identidad
### Objetivo
Que los documentos `identity` reflejen la dirección real elegida y no las alternativas viejas.

### Entregables
- `IDENTITY.md`
- `IDENTITY_DESIGN_BRIEF.md`
- `IDENTITY_IMPLEMENTATION_PLAN.md`
- `IDENTITY_PR_CHECKLIST.md`
- `IDENTITY_ROADMAP.md`

### Estado deseado
Cualquier implementador entiende la dirección correcta al leer la rama `identity`.

---

## Fase 3 — Implementar branding visible mínimo
### Objetivo
Eliminar la mezcla visible entre OpenCode y LightCode en superficies críticas.

### Áreas
- marca visible
- home
- composer
- loading
- títulos visibles
- labels principales

### Estado deseado
El producto visible dice LightCode y usa el vocabulario correcto.

---

## Fase 4 — Implementar Atlas Field como superficie principal
### Objetivo
Traducir el mock validado a la TUI real.

### Trabajo clave
- izquierda = Atlas Index
- centro = Atlas Field
- derecha = Context Panel
- grafo central como navegación real
- clusters y edges con jerarquía
- foco fuerte del thread activo

### Estado deseado
La TUI ya no se siente como un panel clásico de assistant chat.

---

## Fase 5 — Integrar semántica real del producto
### Objetivo
Mapear la interfaz del atlas a entidades reales del sistema.

### Entidades candidatas
- thread actual
- threads relacionados
- working memory
- memory artifacts
- observations
- queued work
- drift o conflictos de naming/estado

### Estado deseado
El Atlas Field deja de ser solo mock visual y empieza a representar datos reales.

---

## Fase 6 — Refinar vocabulario y microcopy
### Objetivo
Alinear el lenguaje visible con la nueva identidad.

### Cambios
- session → thread
- status → telemetry
- memory → memory atlas
- steer current turn → inject signal

### Estado deseado
LightCode suena como LightCode.

---

## Fase 7 — Refinamiento visual
### Objetivo
Pulir jerarquía, color, contraste y densidad sin cambiar la estructura validada.

### Áreas
- color de nodos
- contraste de edges
- legibilidad de labels
- fuerza del nodo activo
- periferia más tenue
- panel derecho más limpio
- panel izquierdo más mínimo

### Estado deseado
La interfaz se siente producto final, no exploración.

---

## Fase 8 — QA de identidad
### Objetivo
Validar consistencia antes de expandir el sistema.

### Revisar
- home
- composer
- Atlas Field
- Context Panel
- labels visibles
- naming
- mezcla de branding
- legibilidad real del campo

### Estado deseado
No quedan superficies importantes que contradigan la identidad.

# Criterio de éxito final

La identidad está conseguida cuando:

- LightCode se percibe como un sistema propio
- el centro del producto es el Atlas Field
- la memoria infinita se siente navegable
- el vocabulario visible es coherente
- la paleta Void Black ya es reconocible
- el producto deja de parecer OpenCode con otro tema
