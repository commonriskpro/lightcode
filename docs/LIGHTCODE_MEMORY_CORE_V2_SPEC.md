# LightCode Memory Core V2 Specification

## 1. Overview

The LightCode Memory Core V1 (commit `efa82ac`) successfully established the foundational database schema, semantic recall patterns, and working memory contracts. However, several critical paths remain disconnected from the active LLM execution loop, rendering parts of the memory system dormant in production.

The goal of LightCode Memory Core V2 is to **wire the existing V1 foundation into the live agent runtime**. V2 focuses entirely on closing integration gaps, ensuring that observations are saved, working memory is dynamically constructed and updated by agents, and semantic recall natively informs the system prompt. We are not rewriting the memory system; we are making it operational.

## 2. Problem Statement (Precise V1 Gaps)

Code audits of the V1 implementation reveal the following critical gaps where memory functionality is either dead code or disconnected from the active agent loop:

### Gap 1: `observeSafe()` is dead code (CRITICAL)

- **Evidence**: `packages/opencode/src/session/om/record.ts` lines 129–140 defines `observeSafe()`, but it is never called anywhere in the codebase.
- **Impact**: In `prompt.ts` (lines 1564–1602), `OMBuf.seal()` and `OM.trackObserved()` advance the system state before `Observer.run()` resolves asynchronously. If `Observer.run()` fails, messages are permanently marked as observed but no record is written to the database. The function signature also currently matches the upsert style, not the buffer+activate path.

### Gap 2: Dream native persistence never triggered (CRITICAL)

- **Evidence**: `packages/opencode/src/dream/index.ts:164` defines `persistConsolidation()`, but it is never called. Furthermore, `dream/daemon.ts:113–119` shows that dream LLM output is never actually read after a session completes.
- **Impact**: The actual consolidation text generated by the dream agent evaporates and never lands in the `memory_artifacts` table.
- **Evidence 2**: `dream/index.ts:95, 120` uses `Engram.ensure()` as a gate, which blocks `autodream` for users who do not have the Engram binary installed.

### Gap 3: Native recall is always empty (CRITICAL)

- **Evidence**: `packages/opencode/src/session/system.ts:165` defines `recallNative()` but passes the project UUID as an FTS5 query. A UUID will never match natural language content.
- **Impact**: The `memory_artifacts` table is always empty in production because nothing currently indexes into it. Even if it did, the FTS5 query is keyword-based (no stemming, no synonyms), and the 300-character content preview is too short to provide useful context to the LLM.

### Gap 4: `Memory.buildContext()` never wired into LLM prompt

- **Evidence**: `packages/opencode/src/memory/provider.ts` contains a correctly defined `buildContext()` method, but it is never called from the `runLoop` in `prompt.ts`.
- **Impact**: The `runLoop` calls `SystemPrompt.observations()` and `SystemPrompt.recall()` individually. Working Memory and Semantic Recall are never composed together into the system prompt context.

### Gap 5: No LLM-facing tool for working memory writes

- **Evidence**: `Memory.setWorkingMemory()` exists in the codebase, but there is no corresponding agent tool exposed to the LLM.
- **Impact**: Agents cannot persist facts or update their working memory during active sessions. The working memory CRUD operations are correct but unreachable from the model.

### Gap 6: Scope maturity is thin

- **Evidence**: Agent, Project, and User scopes are defined in the schema, but there is no operational use.
- **Impact**: No agent writes its own working memory. There is no automatic indexing of project observations into `memory_artifacts`. The global pattern correctly strips private tags, but it is never used in practice.

### Gap 7: Engram gate still blocks autodream

- **Evidence**: `packages/opencode/src/dream/index.ts` `idle()` method awaits `Engram.ensure()`, which fails for users without the Engram binary.
- **Impact**: AutoDream is silently disabled for users without Engram installed. This is a V0 holdover, as the daemon approach (`ensureDaemon`) does not need Engram at all.

## 3. V1 What Was Accomplished

V1 established a robust data layer and foundational abstractions:

- `packages/opencode/src/memory/` contains contracts, schema definitions, working-memory logic, semantic-recall mechanisms, handoff structures, and the main provider.
- 5 new SQLite DB tables: `memory_working`, `memory_artifacts`, `memory_agent_handoffs`, `memory_fork_contexts`, `memory_links`.
- An FTS5 virtual table with sync triggers on `memory_artifacts`.
- `observeSafe()` guard defined in `session/om/record.ts`.
- `persistConsolidation()` helper defined in `dream/index.ts`.
- Native recall path `recallNative()` established in `session/system.ts`.
- Feature flags: `OPENCODE_MEMORY_USE_ENGRAM` (default false), `OPENCODE_DREAM_USE_NATIVE_MEMORY` (default true).
- 33 passing tests in `test/memory/memory-core.test.ts`.

## 4. V2 Goals

V2 aims to achieve operational completeness for the memory system without reinventing it.

### Product Goals:

- **Active Memory:** Agents can explicitly read and write to their working memory during a session.
- **Reliable Consolidation:** Dream daemon successfully persists its summaries so future sessions can recall them.
- **Rich Context:** The LLM receives composed, semantic recall and working memory directly in its system prompt.
- **Dependency Reduction:** Users can utilize native memory and autodream without requiring the legacy Engram binary.

### Technical Goals:

- Wire existing V1 functions into the active execution paths (`prompt.ts`, `dream/daemon.ts`).
- Fix query logic in FTS5 recall to use relevant keywords rather than project UUIDs.
- Expose a `memory_write` tool to the LLM execution loop.
- Remove legacy Engram binary dependencies for the daemon dream process.

## 5. Success Criteria

| Gap       | Success Metric                                                                                                                                                                   |
| --------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Gap 1** | `observeSafe()` is called in the `Observer` flow, ensuring db records are written before marking messages as observed. The signature correctly handles the buffer+activate path. |
| **Gap 2** | `persistConsolidation()` is called after a dream session, and the resulting text is successfully written to the `memory_artifacts` table.                                        |
| **Gap 3** | `recallNative()` uses semantic/keyword queries based on the user's prompt (not the project UUID), and content previews are expanded to provide adequate context.                 |
| **Gap 4** | `Memory.buildContext()` is invoked within `prompt.ts` `runLoop`, properly injecting working memory and recall into the LLM's system prompt.                                      |
| **Gap 5** | A new tool (e.g., `update_working_memory`) is available to the agent, mapping directly to `Memory.setWorkingMemory()`.                                                           |
| **Gap 6** | Basic scope usage is implemented: agents can write to their specific scope, and project observations are automatically indexed.                                                  |
| **Gap 7** | `Engram.ensure()` is removed from the `idle()` check in `dream/index.ts`; autodream runs natively via `ensureDaemon()` without the Engram binary.                                |

## 6. Non-Goals

- **No Vector DBs:** We are sticking with SQLite FTS5 for V2. No external vector databases or embedding models will be introduced.
- **No Schema Changes:** The 5 tables introduced in V1 are sufficient. We will not alter the database schema.
- **No UI Changes:** This is strictly a backend/agent-runtime wiring effort. No CLI UI updates for memory management are included.
- **No Second Memory System:** We are not building a parallel memory system; we are exclusively wiring the V1 implementation.

## 7. Constraints

- **Must Not Break V1 Tests:** All 33 existing tests in `test/memory/memory-core.test.ts` must continue to pass.
- **Additive Only:** Code changes should connect existing disconnected functions. Refactoring should be minimal and focused strictly on wiring.
- **Performance:** Memory recall and context building must not add noticeable latency to the standard prompt loop.

## 8. Migration/Compatibility Expectations

- No database migrations are required, as V1 already created the necessary tables.
- Existing sessions will seamlessly begin utilizing the new native memory paths.
- Feature flags (`OPENCODE_MEMORY_USE_ENGRAM` and `OPENCODE_DREAM_USE_NATIVE_MEMORY`) will be respected, but the default path (`DREAM_USE_NATIVE_MEMORY = true`) will now fully function.

## 9. What "closer to full Engram replacement" means concretely

In V1, Engram was the only functional way to persist session knowledge because the native V1 tables were never written to or read from in the main loop.

"Closer to full Engram replacement" concretely means:

1. Native `memory_artifacts` actually stores the dream summaries (replacing Engram's role in saving session summaries).
2. Native `recallNative()` actually fetches those summaries based on user queries and injects them into the prompt (replacing Engram's role in memory context injection).
3. AutoDream functions purely on the native daemon without checking for the Engram binary installation.
   By achieving this, LightCode's native memory becomes a self-sufficient, closed-loop system capable of persisting and recalling knowledge without external binaries.

## 10. Risks

- **Context Window Bloat:** Injecting both working memory and semantic recall might bloat the system prompt. We must ensure `buildContext()` enforces reasonable limits (e.g., token or character counts) on recalled artifacts.
- **Asynchronous Data Loss:** Fixing `observeSafe()` requires careful handling of async boundaries in `prompt.ts`. If done incorrectly, it could block the main agent loop or still result in dropped observations if the database write times out.
- **FTS5 Limitations:** Keyword-based FTS5 might not perfectly match user intent compared to true semantic search, potentially leading to irrelevant recall. Expanding the content preview mitigates this but requires tuning.
