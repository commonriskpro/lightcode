# LightCode Memory Final Technical Design

## D1 — `addBufferSafe()` (Canonical OM Write Path)

**File**: `packages/opencode/src/session/om/record.ts`

To address the atomicity gap where crashes between buffer addition and tracking lead to duplicate entries, we will introduce `addBufferSafe()`. This function wraps the database operations in a strict transaction.

```ts
export function addBufferSafe(buf: ObservationBuffer, sealAt: number, msgIds: string[]): void {
  Database.transaction(() => {
    // 1. INSERT into ObservationBufferTable (addBuffer equivalent)
    addBuffer(buf)

    // 2. UPDATE ObservationTable to merge msgIds into observed_message_ids
    OM.trackObserved(msgIds)
  })
}
```

_Note_: The seal remains in-memory (`OMBuf.seal(sealAt)`) as it is merely a read-performance hint. If the transaction fails, neither write succeeds, and messages will be safely re-offered at the next threshold crossing.

**Wiring in `prompt.ts`** (lines 1580–1601):
Replace the current sequential block:

```ts
addBuffer({...});
OMBuf.seal(sealAt);
OM.trackObserved(msgIds);
```

With the new transactional approach:

```ts
addBufferSafe(buf, sealAt, msgIds)
OMBuf.seal(sealAt)
```

Replace the generic `"Session observations YYYY-MM-DD"` title with a semantically rich title derived from the actual task or content. Also, prioritize `reflections` over raw `observations`.

```ts
// Prioritize reflections for the core content
const content = finalObs.reflections ?? finalObs.observations

// Generate a rich, searchable title

```
