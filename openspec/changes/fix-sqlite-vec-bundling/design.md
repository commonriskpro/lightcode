# Design — Fix `sqlite-vec` Bundling in Compiled Binary

## Investigation

### Reproduction

```
$ bun run build --single
... build succeeds ...
Smoke test passed: 1.4.1   # ← lies, this is the parent test from a stale build

$ ./dist/opencode-darwin-arm64/bin/lightcode
Error: Unexpected error, check log file at .../2026-04-08T193240.log for more details
Cannot find module 'sqlite-vec-darwin-arm64/vec0.dylib' from '/$bunfs/root/src/index.js'
```

Log tail:

```
INFO  service=db path=/Users/.../lightcode.db opening database
[crash]
```

The crash happens at `packages/opencode/src/storage/db.bun.ts:37`:

```ts
sqliteVec.load(sqlite)
```

### Root cause

`sqlite-vec`'s loader (`node_modules/.bun/sqlite-vec@0.1.9/node_modules/sqlite-vec/index.mjs`):

```js
function getLoadablePath() {
  // ...
  const packageName = platformPackageName(platform, arch)
  const loadablePath = fileURLToPath(
    import.meta.resolve(packageName + "/" + ENTRYPOINT_BASE_NAME + "." + extensionSuffix(platform)),
  )
  return loadablePath
}

function load(db) {
  db.loadExtension(getLoadablePath())
}
```

Two layers of failure inside the compiled binary:

1. **`import.meta.resolve()` runs in `$bunfs`.** The `sqlite-vec` package source is bundled
   into `$bunfs/root/...`, so its `import.meta` URL is a `bunfs://` URL. Resolution targets
   `$bunfs/root/sqlite-vec-darwin-arm64/vec0.dylib`, which does not exist because Bun's
   `--compile` only bundles JS/TS modules detected by static analysis, not native binaries
   in dependencies.

2. **`dlopen()` cannot read from `$bunfs`.** Even if step 1 returned a `$bunfs` path,
   `Database.loadExtension()` ultimately calls `dlopen()` (or `LoadLibrary` on Windows),
   which is a syscall that only understands real OS filesystem paths. The `$bunfs` is a
   userspace virtual filesystem in the Bun runtime — invisible to the kernel.

### What works in `bun run dev`

In dev mode, `import.meta.resolve()` returns the real `node_modules/.bun/sqlite-vec-darwin-arm64@0.1.9/...`
path on disk. `dlopen()` reads it directly. No issue.

### Available native libraries (already installed)

```
node_modules/.bun/sqlite-vec-darwin-arm64@0.1.9/.../vec0.dylib
node_modules/.bun/sqlite-vec-darwin-x64@0.1.9/.../vec0.dylib
node_modules/.bun/sqlite-vec-linux-arm64@0.1.9/.../vec0.so
node_modules/.bun/sqlite-vec-linux-x64@0.1.9/.../vec0.so
node_modules/.bun/sqlite-vec-windows-x64@0.1.9/.../vec0.dll
```

The `bun install --os="*" --cpu="*" @opentui/core` line in `build.ts:182–184` already pulls
all platform-specific optional deps into `node_modules`. `sqlite-vec` declares the same
optional deps in its `package.json`:

```json
"optionalDependencies": {
  "sqlite-vec-darwin-x64": "0.1.9",
  "sqlite-vec-linux-x64": "0.1.9",
  "sqlite-vec-darwin-arm64": "0.1.9",
  "sqlite-vec-windows-x64": "0.1.9",
  "sqlite-vec-linux-arm64": "0.1.9"
}
```

So the cross-platform `.dylib` / `.so` / `.dll` files are already on disk during the build.
We just need to embed the right one per target.

## Decisions

### D1 — Embed via `with { type: "file" }`, extract at runtime

**Decision**: Use Bun's `import x from "./foo.dylib" with { type: "file" }` to embed the native
library inside the compiled binary as an asset. At runtime, copy the asset bytes to a real
filesystem path under the user's cache directory and call `loadExtension()` against that path.

**Why**:

- `with { type: "file" }` is the only documented way to embed arbitrary binary assets inside a
  `bun build --compile` output.
- `dlopen()` requires a real filesystem path. Extraction is mandatory, not optional.
- The pattern is already used in `script/build.ts:79` for the embedded web UI assets.

**Alternatives rejected**:

- **Sidecar `.dylib` next to the binary**: defeats single-file distribution. Requires release
  scripts to bundle a directory instead of a single binary, breaks `gh release upload` of a
  single asset, and a user copying just `lightcode` to `~/bin` would silently break.
- **`require.resolve()` at runtime**: still hits `$bunfs`, same crash.
- **Cross-compile our own `.dylib`**: massive scope explosion, requires CI matrix for 5
  platforms, and `sqlite-vec` upstream already does this work.

### D2 — Per-target virtual file generated by `build.ts`

**Decision**: For each of the 12 build targets, `build.ts` generates a `vec-loadable.gen.ts`
file with the absolute path to the matching native library, and passes it to
`Bun.build({ files: { "vec-loadable.gen.ts": <source> } })`. `db.bun.ts` imports that virtual
file by name and gets the embedded asset's `$bunfs` path.

**Why**:

- We compile 12 targets in one `build.ts` invocation. Each target needs a different binary
  embedded. A static `import` in `db.bun.ts` would pin a single platform.
- `Bun.build` already accepts virtual entrypoints via the `files` map (used by
  `embeddedFileMap` for the web UI).
- The virtual file is a one-line module: `import dylib from "<abs path>" with { type: "file" }; export default dylib`.
  Trivial to generate, no template needed.

**Alternatives rejected**:

- **Conditional imports inside `db.bun.ts`** (`if (process.platform === "darwin") import "..."`):
  Bun's static analyzer would still try to resolve all branches, embedding all 5 binaries in
  every target.
- **A `define` injection** of the absolute path string: doesn't actually import the asset, so
  Bun would not embed it. Only `import x from y with { type: "file" }` triggers embedding.
- **Build-time `cp` of the `.dylib` into `dist/<target>/bin/`**: this is the sidecar approach
  again. Rejected by D1.

### D3 — Cache extracted `.dylib` under `XDG_CACHE_HOME/lightcode/sqlite-vec/vec0-<hash>.<ext>`

**Decision**: On startup, hash the embedded asset's bytes (xxhash, already a dep), check for
`<cache>/vec0-<hash>.<ext>`, and write only if missing. Use `XDG_CACHE_HOME` if set, otherwise
`~/.cache/lightcode/sqlite-vec/`. Fall back to `os.tmpdir()/lightcode-vec0-<hash>.<ext>` if
the cache dir is not writable.

**Why**:

- **Hash-based filename** = automatic invalidation when `sqlite-vec` is upgraded. Old files
  remain in cache (cleanup is a future change), but new versions get a new filename.
- **Skipping rewrite when the file exists** keeps startup fast: `fs.existsSync` is microseconds,
  the actual write is ~10ms.
- **Cache dir over `/tmp`** survives reboots, avoids re-extracting on every startup of a system
  that aggressively cleans `/tmp`.
- **`/tmp` fallback** handles read-only `$HOME` (CI runners, sandboxed envs).

**Alternatives rejected**:

- **Always extract to `/tmp` with random suffix**: re-extracts on every startup, wastes disk.
- **Extract next to `process.execPath`**: requires write access to wherever the user installed
  the binary (probably `/usr/local/bin/`, owned by root).
- **In-memory `dlopen` via `memfd_create`**: Linux-only, not portable to macOS/Windows.

### D4 — Dev mode fallback to `sqliteVec.load(sqlite)`

**Decision**: Wrap the `vec-loadable.gen.ts` import in a `try`. If it throws (file does not
exist in `bun run dev`), fall back to `sqliteVec.load(sqlite)`.

**Why**:

- `bun run dev` does not invoke `build.ts`, so `vec-loadable.gen.ts` is not generated.
- The original `sqliteVec.load(sqlite)` works fine in dev because `import.meta.resolve()`
  resolves to a real disk path.
- A single `try` wrapper preserves both code paths without conditionals on `process.env.NODE_ENV`
  or any other indirect signal.

**Alternatives rejected**:

- **Generate the file once and commit it**: pollutes the repo with a generated artifact and
  pins the path to one platform.
- **Make `vec-loadable.gen.ts` a real file checked into git**: same problem.

### D5 — Use `await import()` (dynamic) instead of static `import`

**Decision**: In `db.bun.ts`, load the generated module via:

```ts
const vec = await import("./vec-loadable.gen.ts").catch(() => null)
```

**Why**:

- A static `import` triggers TypeScript and Bun resolution at file-parse time. With no
  `vec-loadable.gen.ts` on disk during `bun run dev` or `bun typecheck`, both would error.
- `await import()` defers resolution to runtime and the `.catch(() => null)` makes the missing
  file a soft failure.
- `init()` in `db.bun.ts` is already called at runtime from a place that can be made async
  (only one caller, `Storage` initialization).

**Side effect**: `init()` becomes `async`. Callers must `await` it.

### D6 — `init()` becomes async

**Decision**: Change `export function init(path: string)` to `export async function init(path: string)`
and update the only caller (`storage/db.ts` re-export consumed by Drizzle setup).

**Why**:

- D5 forces async because we use `await import()`.
- The `Database` instance must be returned only after `loadExtension()` succeeds, so the whole
  chain serializes.

### D7 — Extension naming per platform

**Decision**: A small helper inside `db.bun.ts`:

```ts
const ext = process.platform === "win32" ? "dll" : process.platform === "darwin" ? "dylib" : "so"
```

Used both for the cached filename and the asset extension.

**Why**:

- Mirrors `sqlite-vec`'s own `extensionSuffix()` helper.
- We don't need to import that helper because we only run on the host platform at runtime —
  the build-time per-target choice is made by `build.ts`, not `db.bun.ts`.

### D8 — Hash via xxhash-wasm (already a dep)

**Decision**: Use `xxhash-wasm` (in `package.json` dependencies) for the content hash:

```ts
const xx = await xxhash()
const hash = xx.h64ToString(bytes)
```

**Why**:

- Already a dep, no new package.
- 64-bit hex is short enough for a filename.
- Faster than `crypto.createHash("sha256")` and the threat model doesn't need cryptographic
  collision resistance — just content-uniqueness for cache invalidation.

## Data flow

### Build time (`bun run build`)

```
script/build.ts
  for target in [darwin-arm64, darwin-x64, linux-arm64, linux-x64, windows-x64, ...]:
    1. Resolve native lib path:
       node_modules/.bun/sqlite-vec-<os>-<arch>@0.1.9/.../vec0.<ext>
    2. Generate virtual file source:
       import dylib from "<abs path>" with { type: "file" }
       export default dylib
    3. Bun.build({ files: { "vec-loadable.gen.ts": source }, ... })
       → Bun reads the absolute path, embeds the .dylib bytes into $bunfs
       → vec-loadable.gen.ts becomes a JS module returning a $bunfs path string
    4. The compiled binary contains:
       - all JS/TS code
       - the embedded web UI files
       - the per-target native vec0 library
```

### Runtime (compiled binary, first startup)

```
init(path)
  1. const vec = await import("./vec-loadable.gen.ts").catch(() => null)
  2. if vec exists:
       const bunfsPath = vec.default              // e.g. "/$bunfs/root/.../vec0.dylib"
       const bytes = await Bun.file(bunfsPath).bytes()
       const hash = xxhash64(bytes)
       const cacheDir = $XDG_CACHE_HOME/lightcode/sqlite-vec
       const cached = cacheDir/vec0-<hash>.<ext>
       if (!exists(cached)) {
         mkdir cacheDir
         write cached <- bytes
         chmod 0755 cached
       }
       sqlite.loadExtension(cached)
     else:
       sqliteVec.load(sqlite)                     // bun run dev fallback
```

### Runtime (compiled binary, subsequent startups)

```
init(path)
  1. await import works → bunfsPath
  2. read bunfs bytes → hash → cache lookup
  3. cached file exists → skip write
  4. sqlite.loadExtension(cached)
```

The expensive step is always the bunfs read + hash (~5–10ms). The actual disk write happens
once per version per host.

## Out-of-scope risks (documented, not fixed)

- **Stale cached `.dylib` files**: hash-based filenames mean old files persist forever. Future
  cleanup change can prune `vec0-*.dylib` not matching the current hash on startup.
- **Cache dir collisions across multiple lightcode versions**: not a problem because the hash
  is content-based, so two versions shipping the same upstream `sqlite-vec` reuse the same
  file.
- **Windows path separators**: `sqlite.loadExtension()` accepts forward slashes on Windows in
  Bun. Tested by Bun's own SQLite suite.
- **macOS Gatekeeper / quarantine attribute**: extracted `.dylib` may need `xattr -d
com.apple.quarantine` if downloaded from the internet. The source bytes come from the
  binary itself (already through Gatekeeper if installed via Homebrew), so the extracted
  copy inherits the parent's status. Untested edge case for direct binary downloads from
  GitHub releases — to be validated post-merge.

## Open questions (none blocking)

- Should we also handle `XDG_CACHE_HOME` being set to an empty string (vs. unset)? Linux spec
  says treat empty as unset. Will follow that.
